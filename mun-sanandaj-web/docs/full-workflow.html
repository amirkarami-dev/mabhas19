Full Work Flow — mun-sanandaj-web

How the whole service works, from start to end, in simple words.

In one line: two small robots wake up on their own, read a list of building projects from the city's database, and send each project's report and engineer map to the Sanandaj city system. The web page just shows what the robots did.

1The parts

These are the pieces that work together. Each one has one job.

Web / API (ours) Robots (background jobs) Databases Outside systems
The web page (React app)
What you open in the browser. It only reads and shows data. It does not send anything to the city itself.
Login server (IdP)
Checks your user name and password. Only admin users may open the page. (See the login flow doc.)
The API (backend)
The brain. It holds the robots, talks to the databases, and gives the web page its data.
Robot 1 — Engineer Report
Sends the supervising engineer's PDF report to the city.
Robot 2 — Engineer Map
Sends the list of engineers (the "map") for each project to the city.
KurdNezam database
The city's own database. We only read from it. It gives us the list of projects and the engineers.
City system (mahyapardaz)
The web service we send data to: saveEngineerReport, saveEngMap, and addEngineer.
PDF server
Holds the ready-made PDF file for each project. We download it and turn it into text (base64) before sending.
Our database (logs)
Where we write down every run and every try, so the web page can show the history.

2The big picture

This loop runs on its own. Nobody has to press a button.

Step 1 · Timer
Every 12 hours the two robots wake up
They also run once right away each time the server starts (each new deploy). You can also press Run now on the page to start one at any time.
Step 2 · Read the list
Ask the city database for the projects
Each robot calls the saved question WebS_GetListRepToShahrdari. It returns rows like Peygiri (tracking code), ProjectNo, and ReqId.
for each row
Step 3 · Do the job
Send each project to the city
The robot goes through the rows one by one and sends each one (see the two jobs below).
Step 4 · Write it down
Save the result of every try
For each project it writes one log row: success or failed, with the error text if it failed.
later
Step 5 · Show it
You open the page and see the results
The web page reads the saved rows and shows counts, a chart, and a table.

3Job 1 — send the Engineer Report

What Robot 1 does with one project row.

Step 1
Was this project already sent OK before?
If the last try for this Peygiri was success, skip it. We do not send the same thing twice.
not sent yet
Step 2
Get the PDF file
Download the PDF for this project from the PDF server and turn it into text (base64).
Step 3 · Check
No PDF yet?
If the PDF is not ready (not found), mark failed: pdf not found and stop. The robot will try again on the next run.
PDF is ready
Step 4 · Send
Send the PDF to the city
Call saveEngineerReport with the project number and the PDF text.
City says OK ✓
success
Save the city's tracking number (peigiri) with the log row.
City says error ✗
failed
Save the error text. Try again on the next run.

4Job 2 — send the Engineer Map

Robot 2 is like Robot 1, but it also has an extra "add the engineer first, then try again" step.

Step 1
Already sent OK before? → skip
Same rule as Job 1: do not send the same project twice.
Step 2 · Read engineers
Ask the database who the engineers are
Call the saved question WebS_GetReportFullInfo with the project's Peygiri. It returns each engineer's name, national code, role, and so on.
Step 3 · Send map
Send the engineers + PDF to the city
Call saveEngMap with the list of engineers and the PDF text (same PDF step as Job 1).
read the answer
Step 4 · What did the city say?
Three possible answers
The next three boxes show each case.
Answer A · all good
success
The city accepted the map. Save the tracking number and finish.
Answer B · "engineer not found" (مهندس یافت نشد)
Add the engineer first, then try one more time
1) The city does not know this engineer yet.
2) Call addEngineer to create that engineer in the city system.
3) Call saveEngMap again, just once more.
→ If now OK → success (and we note which engineers we added). Still not OK → failed, try again next run.
Answer C · a different error
failed
We do not add an engineer for this case. Save the error and try again on the next run.

5What you see on the page

Two screens. Both are read-only views of the saved rows.

Dashboard (داشبورد)
Two cards, one per robot, each with OK / failed / total counts. A chart of the last runs. A table of the last run's rows. A Run now (اجرای فوری) button per robot. While a robot is working, a small blue dot pulses. The page refreshes itself every few seconds.
Logs (تاریخچه)
One big table of every try, with pages. You can filter by robot type, by success/failed, or search by tracking code or project number.

6What we save

Two tables in our own database. We never change old rows — we only add new ones.

TableOne row means…Holds
mun_sync_runs one time a robot ran which robot, start & end time, running / done / failed, and the OK / failed / total counts.
mun_report_logs one try to send one project the tracking code, the status, the try number, the city's tracking number, and the error text if any. Kept forever.
Safe by default: if the city database address is not set on the server, the robots do not start at all. The feature is simply turned off, and the Run now button returns a clear message instead of an error. A project that failed will be tried again on the next run — the robots keep trying until it works.
