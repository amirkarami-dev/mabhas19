# Dev status / roadmap board тАФ design

> **Date:** 2026-06-25 ┬╖ **Status:** approved (mockup greenlit) ┬╖ **Author:** Amir + Claude
> **Deliverable:** a static HTML5 development status board for the Mabhas19 portal.

## Goal

A single, self-contained HTML5 page that shows the **full development status + todo backlog**
of every portal service, split by status (**not-started / in-progress / done**, plus
**blocked**), so the owner can see at a glance what is built and what still needs building тАФ
with special emphasis on the unbuilt work needed to finish the **analytics ("hero analytic")
platform**.

## Decisions (from brainstorming)

| Question | Decision |
|---|---|
| Data model | **Editable `roadmap.json`** rendered by the page (no backend, no drag-persist). |
| Service scope | **All live + planned**: `analytic`, `mabhas19`, `auth`, `storage`, `mobile` (live), `gateway`, `portal` (planned). |
| Language / direction | **English, LTR** (internal dev/PM tool). |
| Hosting (`develop.myceo.ir`?) | **Do not** build a heavyweight new service yet, and don't name it `develop` (reads like a product staging env). Build as static files now; if promoted to prod, deploy as a **gated static site** (auth/IP-restricted, DNS-only, no CDN тАФ per ADR-019) at **`status.myceo.ir`**, ideally a tiny nginx service on the analytics box. Promote to a real `develop.` internal portal only if it grows beyond a roadmap. |

## Architecture

Three static files under `roadmap/`, zero build step, zero runtime dependencies:

- `index.html` тАФ HTML + CSS + vanilla JS. Emerald theme, dark/light toggle (localStorage +
  `prefers-color-scheme`), responsive (3 тЖТ 2 тЖТ 1 columns). Fetches `roadmap.json`.
- `roadmap.json` тАФ the editable source of truth (schema below).
- `README.md` тАФ how to edit + how to deploy.

### Data flow

`roadmap.json` тЖТ `fetch()` тЖТ render service selector тЖТ on select, render the kanban
(status columns) for that service тЖТ search/area/priority filters re-render client-side.
Per-service progress = `done/total`; overall progress aggregated in the header.

### Schema (per item)

`{ id, title, desc?, status, area, priority, effort, tags?, notes? }` grouped under each
service `{ id, name, domain, kind: "live"|"planned", items[] }`. Enums: `status` тИИ
{not-started, in-progress, done, blocked}; `area` тИИ {backend, frontend, devops, infra, mobile,
docs}; `priority` тИИ {high, med, low}; `effort` тИИ {XS, S, M, L}.

### UI components

- **Header** тАФ title, last-updated, overall progress %, theme toggle.
- **Service selector** тАФ pill per service with per-service %; `planned` pills dashed + tagged.
- **Service meta** тАФ name ┬╖ domain, live/planned pill, progress bar.
- **Toolbar** тАФ search, area filter (populated per service), priority filter, reset.
- **Board** тАФ status columns with counts; cards show title, desc, area/priority/effort badges;
  `blocked` items get a red column (shown only when present) and a badge.

### Content (the `roadmap.json` data)

Generated by a fan-out inventory workflow (`dev-roadmap-inventory`) тАФ one reader agent per
service classifies real work items as done / in-progress / not-started from the actual code,
docs (`architecture-review-2026.md` deferred table, the `report-web-v1` memory, ADRs), and the
parked API-gateway brainstorm. The board is curated, not live тАФ update by editing `roadmap.json`.

## Caveats

- `file://` blocks `fetch()`; the page detects this and prints the `npx serve roadmap` one-liner.
  Works directly when nginx-served on prod.
- The board reveals internal/unfinished work тЖТ must be gated wherever it's hosted.

## Out of scope (v1)

Drag-to-reorder + persistence, live GitHub/API backing, multi-user editing, auth inside the
page (gating happens at the proxy), bilingual toggle.
