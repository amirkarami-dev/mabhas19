Mabhas19 Platform — System & Developer Guide

This guide documents Mabhas19 (مبحث ۱۹), a full-stack platform for building-energy assessment under Iran's National Building Code, Section 19, Appendix 5 (5th edition). It is both a system workflow guide and a developer guide. The reader is the development team.

1. Overview — What is Mabhas19

Section 19 (in Persian, "Mabhas 19") is the part of Iran's National Building Code that covers energy saving in buildings. Appendix 5 of its 5th edition defines a scoring method to check how energy-efficient a building is.

The platform lets an engineer:

  • Create a building project (by hand or by importing from an external registry).
  • Run six checklists that score the building's energy design.
  • Get one total score (maximum 831 points) plus per-section scores.
  • Generate a Persian PDF report of the result.

The platform has three parts that all share one scoring engine:

  • A web app (Next.js) for desktop use.
  • A mobile app (Expo / React Native) for phones.
  • A backend API (.NET) that stores all data and is the system of record.
Key idea: The six-checklist scoring runs in the frontend (web and mobile), using a shared TypeScript package. The backend stores inputs, results, and scores, and builds the PDF. Scoring rules change in the shared package, not in the backend.

6. Backend (Clean Architecture)

The backend follows Clean Architecture with four layers. Dependencies always point inward: outer layers depend on inner ones, never the reverse.

graph LR
  WEB["Web
(Minimal API endpoints)"] --> APP
  INFRA["Infrastructure
(EF Core, MinIO, SMS, Identity)"] --> APP
  APP["Application
(CQRS use cases, validators, mapping)"] --> DOM
  DOM["Domain
(entities, enums, calculators)"]
  WEB --> INFRA
  • Domain — entities, enums, constants, and the Section 19 calculators (BuildingGroupCalculator, ClimateData). No dependencies on other layers.
  • Application — CQRS (Command Query Responsibility Segregation) use cases via MediatR, FluentValidation validators, AutoMapper profiles, and service interfaces. Depends on Domain only.
  • Infrastructure — concrete service implementations, EF Core/SQL Server, ASP.NET Identity. Depends on Application + Domain.
  • Web — Minimal API endpoints. Each IEndpointGroup class is auto-mapped at /api/{ClassName}.

MediatR pipeline behaviors

Every request passes through these cross-cutting behaviors, in order:

  1. LoggingBehaviour — logs all requests.
  2. UnhandledExceptionBehaviour — catches and logs exceptions.
  3. AuthorizationBehaviour — enforces [Authorize] and role/policy checks.
  4. ValidationBehaviour — runs FluentValidation validators.
  5. PerformanceBehaviour — logs slow requests.

Section 19 domain calculators

BuildingGroupCalculator.Calculate(area, floors, units) classifies a building into one of 7 groups (A, B, B+, C, C+, C++, D). It is a faithful port of the legacy JS calcBuildingGroup logic. Sequential rules:

  1. D: floors > 10 OR area > 5000.
  2. C / C+ / C++: floors 6–10 OR area 2001–5000. If units > 30: C+ when area < 3000, else C++; otherwise C.
  3. B / B+: floors 3–5 OR area 601–2000. If units > 30: B+; otherwise B.
  4. A: floors ≤ 2 AND area ≤ 600.
  5. Default: B.

Persian labels: A→الف, B→ب, B+→ب+, C→ج, C+→ج+, C++→ج++, D→د.

ClimateData holds static reference data: 6 climate zones (1, 2, 3A, 3B, 4, 5), a city→climate map for 31 Iranian cities (GetCityClimate defaults to "4"), R-value (thermal resistance) targets per climate, U-value (transmittance) limits for transparent elements, and SHGC (Solar Heat Gain Coefficient) limits.

Database migrations & seeding

  • Migrations are applied automatically on API startup by ApplicationDbContextInitialiser.InitialiseAsync() (calls MigrateAsync()).
  • Seeding ensures both roles exist (Administrator, User).
  • An admin user is created from config (AdminUser:Email default admin@mabhas19.myceo.ir, AdminUser:Password) with EmailConfirmed = true and the Administrator role.
  • The admin user gets an Enterprise subscription with MaxProjects = 1000.

Add a migration with:

dotnet ef migrations add <Name> --project src/Infrastructure \
  --startup-project src/Web --output-dir Data/Migrations

Subscription quota

  • Free plan default cap: 5 projects (DefaultMaxProjects = 5).
  • ISubscriptionService.EnsureCanCreateProjectAsync(userId) runs inside CreateProject and ImportProject.
  • It counts projects owned by the user. If count >= MaxProjects, it throws a ValidationException on the Subscription field (Persian message: "به سقف تعداد پروژه‌های مجاز رسیده‌اید").
  • It also checks IsActive.

7. The Assessment Scoring Engine

The shared package @mabhas19/assessment-core (in packages/assessment-core/) is the single source of truth for all scoring. It is pure TypeScript with no framework dependencies, so both the web and mobile apps import the exact same logic. This means scores never disagree between platforms.

Total maximum score: 831 points (exported as TOTAL_MAX_SCORE).

Scoring functions

Each checklist has a pure function (signature roughly score<Checklist>(input, climateCode?) → ToolScore):

  • scoreEnvOpaque — checks opaque R-values against climate targets; thermal bridges and shading. Returns extra flags like envelopePass, bridgePass.
  • scoreEnvTrans — computes window U-values from glass/gas/frame layers; checks against climate U and SHGC limits.
  • scoreMech — pass/fail logic over mechanical rows (HVAC, heating, cooling, hot water, controls).
  • scoreElec — electrical scoring (lighting, PV, demand response, controls).
  • scoreMonitoring — building monitoring and sub-metering.
  • scoreIntegrated — integrated building management.

A dispatcher scoreTool(toolKey, input, climateCode) routes input to the right function and returns { score, maxScore } (zeros for an unknown toolKey).

Exported data

  • src/data/climate — zones, city map, R/U/SHGC targets.
  • src/data/sections — the six checklists and their max scores (ASSESSMENT_SECTIONS, ALL_TOOLS).
  • Reference databases: envOpaqueDb, envTransDb, mechDb, elecDb, monitoringDb, integratedDb.
  • Helpers: calcBuildingGroup({ area, floors, units }), getOpaqueTargetR, calcWindowU, getTransShgcLimit, getGroupIndex.

Key data types: ToolScore = { score, maxScore }; BuildingMeta = { totalArea, floorCount, unitCount, city?, usage? }; ToolResult = { toolKey, score, maxScore, details? }.

9. Authentication & Roles

There are three sign-in methods. All of them end with the API issuing an Identity bearer token (an access token plus a refresh token). The OTP and Google flows set signInManager.AuthenticationScheme = IdentityConstants.BearerScheme before calling SignInAsync.

Roles: Administrator (manage users + subscriptions, access /api/Admin/*) and User (default). GET /api/Users/me returns { roles, isAdmin }.

Password login

sequenceDiagram
  participant C as Client (web/mobile)
  participant A as API (Identity)
  C->>A: POST /api/Users/login (email, password)
  A->>A: Validate credentials
  A-->>C: { accessToken, refreshToken }
  C->>A: GET /api/Users/me (Bearer accessToken)
  A-->>C: CurrentUserDto { roles, isAdmin }

Mobile OTP (one-time password)

sequenceDiagram
  participant C as Client
  participant A as API
  participant O as OtpService (cache)
  participant S as SMS relay
  C->>A: POST /api/Auth/otp/request (phoneNumber)
  A->>O: Generate N-digit code, store key otp:{phone} (TTL)
  A->>S: SendAsync(phone, "code: NNNN")
  S-->>C: SMS delivered to phone
  C->>A: POST /api/Auth/otp/verify (phone, code)
  A->>O: FixedTimeEquals(code); delete on match
  A->>A: Create user if new; SignInAsync (Bearer)
  A-->>C: { accessToken, refreshToken }

OTP details: code length and time-to-live (TTL) come from OtpOptions. Codes are stored in IDistributedMemoryCache and compared with CryptographicOperations.FixedTimeEquals to stop timing attacks. OtpOptions.LogCode can log the code in development only.

12. Assessment Workflow

The whole flow, from creating a project to a finished PDF:

flowchart TD
  A["Create project (manual)
or Import from NezamMohandesi"] --> B["Climate zone auto-set from city"]
  B --> C["Open assessment
/projects/{id}/assessment"]
  C --> D["Fill the 6 checklists"]
  D --> E["Live scoring via
@mabhas19/assessment-core (scoreTool)"]
  E --> F{"Edit more?"}
  F -- yes --> D
  F -- no --> G["Save: PUT /api/Projects/{id}/assessment
inputJson + resultJson + totalScore + maxScore"]
  G --> H["Generate report: POST /api/Projects/{id}/report"]
  H --> I["QuestPDF builds Persian PDF"]
  I --> J["Store PDF in MinIO (StorageKey)"]
  J --> K["Return presigned download URL (valid 1 hour)"]

Step by step:

  1. The user creates a project by hand, or imports one from the external registry. The subscription quota is checked first.
  2. The climate zone is set automatically from the project's city.
  3. The user opens the assessment workspace. The app loads the project meta and any saved inputJson/resultJson.
  4. The user fills in the six checklists. Each one shows a live score / maxScore badge.
  5. Scoring runs in the browser/app through scoreTool from the shared package. The total is shown out of 831.
  6. On Save, the client posts inputJson, resultJson, totalScore, and maxScore. The backend stores them (nvarchar(max) plus denormalized score columns).
  7. On report request, the backend renders a Persian PDF with QuestPDF, uploads it to MinIO, records an AssessmentReport, and returns a presigned URL valid for one hour.

13. Internationalization

  • Two locales: fa-IR (Persian, default, RTL) and en-US (English, LTR).
  • Web uses next-intl with localePrefix: "as-needed": fa is served at /, English at /en/.... localeDetection is off.
  • Message files: web/messages/fa.json and web/messages/en.json.
  • RTL: dir="rtl" when the locale is fa. A .flip-x utility mirrors SVGs; CSS logical properties (start/end) handle spacing in both directions.
  • Use the locale-aware Link/useRouter from @/i18n/navigation, not next/link.
  • Mobile hardcodes fa as default with an English fallback for non-Persian devices.

14. PDF Reports & File Storage

PDF generation (QuestPDF)

  • QuestPdfReportGenerator.GenerateAssessmentReport(model) returns a byte[].
  • A4 page, right-to-left layout, Persian fonts (Vazirmatn). Header: "گزارش ارزیابی جامع انرژی ساختمان" (Comprehensive Building Energy Assessment Report).
  • Content: a project metadata table, a per-section scoring table, and a total score box. Footer shows the timestamp and brand.
  • Sections covered: opaque envelope, transparent envelope, mechanical, electrical, monitoring, integrated management.
  • QuestPDF uses its free community license.

File storage (MinIO)

  • MinioFileStorage implements IFileStorage: PutAsync, GetAsync, DeleteAsync, GetPresignedUrlAsync(key, expiry).
  • The presigned URL is time-limited (1 hour for reports). In production, the host is rewritten to the public endpoint so browsers can open it.
  • Config (MinioOptions): endpoint, access key, secret key, bucket, and a PublicEndpoint for proxy-behind deployments.

17. Glossary

TermMeaning
Section 19 (Mabhas 19)The part of Iran's National Building Code that covers energy saving in buildings. Appendix 5 (5th ed.) defines the scoring method this platform implements.
OTPOne-Time Password. A short code sent by SMS to log in.
OKLCHA modern CSS color space (Lightness, Chroma, Hue) used for the web design tokens.
nvarchar(max)SQL Server's large variable-length text column type. Stores the assessment JSON (InputJson and ResultJson).
CQRSCommand Query Responsibility Segregation. Splits write operations (commands) from read operations (queries). Implemented with MediatR.
RTLRight-To-Left text direction, used for Persian.
Presigned URLA temporary, signed link to a file in object storage (MinIO) that works without login for a limited time.
Bearer tokenAn access token sent in the Authorization: Bearer ... header to prove who the user is.
Building group (A–D)A Section 19 size class (A small to D large) computed from area, floor count, and unit count.
R-valueThermal resistance. Higher means better insulation. Checked for opaque surfaces.
U-valueThermal transmittance. Lower means less heat loss. Checked for windows/doors.
SHGCSolar Heat Gain Coefficient. How much solar heat passes through glazing.
MinIOAn S3-compatible object storage server. Stores the generated PDF reports.
QuestPDFA .NET library used to build the Persian PDF reports.
TraefikA reverse proxy that routes traffic to the web, API, and S3 domains and handles TLS.
NezamMohandesiIran's Building Engineering System — the external registry projects can be imported from.
EASExpo Application Services — the cloud build service used to produce the mobile APK/AAB.
DenormalizeStoring a value (like TotalScore) in its own column even though it could be computed, so queries are faster.
